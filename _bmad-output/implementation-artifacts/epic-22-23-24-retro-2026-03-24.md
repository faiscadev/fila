## What Went Well

### Implementation Quality
Seven stories, seven clean PRs, zero Cubic findings. Config changes and mechanical type conversions are low-risk. The code quality was the smoothest of any epic sequence.

### Backward Compatibility
Every optimization preserved backward compatibility: `bytes::Bytes` has the same wire format as `Vec<u8>`, `shard_count=1` is identical to pre-sharding, `BatchEnqueue` is additive (single enqueue still works), delivery batching expands transparently in the SDK bridge.

### Code Review Catches
- WAL durability regression in 22.1 (`manual_wal_flush` default unsafe for single-node)
- Bridge task cancellation bug in 23.3 (caught by existing e2e test)

### 64KB and Tail Latency Wins
The zero-copy work (24.1) delivered 3.3x improvement for large messages — beating the research prediction. The tail latency improvement (-91% p99.9) was an unpredicted benefit, likely from write coalescing reducing contention spikes.

---

## What Didn't Go Well

### Research Predictions Wildly Off
The research based targets on what Kafka/NATS/EMQX achieve (append-only logs, sendfile(), io_uring) — architecturally different systems. Fila's RocksDB + DRR + per-message Lua hook architecture has a fundamentally lower throughput ceiling. Predictions should have been based on profiling the actual system.

### No Profiling Before Optimizing
All 3 epics optimized based on theory without profiling to confirm where time is spent. The DRR O(1) fix is the clearest example — predicted 3x at 10K keys, actual was flat, suggesting the O(n) scan wasn't the real bottleneck.

### Benchmarks Don't Exercise New Paths
The self-benchmark suite doesn't call `batch_enqueue`, doesn't use multi-queue sharding, and tests single-producer by default. The primary throughput lever (batching) is unmeasured by the standard suite.

### Batch Operations Are Rust-Only
All 5 external SDKs (Go, Python, JS, Ruby, Java) have no `BatchEnqueue` RPC, no delivery batching, and outdated proto files. Story 23.1 explicitly scoped them out. The primary throughput optimization is unavailable to most SDK users.

### Core Path Regressions
FIFO (-7%), Fair (-6%), and Lua (-9%) throughput regressed. Larger RocksDB write buffers and sharding overhead add per-message cost that outweighs copy savings for small messages.

### Memory and Disk I/O Costs
+79% memory (from 128MB write buffers) and +362% disk I/O (larger buffers = more data in-flight before compaction). These are conscious tradeoffs from the RocksDB tuning, but significant.

### All Benchmarks Deferred
Epic 21 retro action item #2 said "each optimization must be benchmarked." All 7 stories deferred measurement. The numbers Lucas provided are the first post-optimization measurements.

## Key Takeaways

1. **Profile before optimizing.** Research predictions based on other systems' architectures are unreliable. Fila's throughput ceiling is determined by its own hot path, not by what Kafka achieves with a fundamentally different design.

2. **Benchmarks must exercise new code paths.** Building `batch_enqueue` without benchmarking it means we don't know if it works. Adding batch scenarios to fila-bench is part of Epic 27.

3. **SDK feature parity matters.** Optimizations that only exist in the Rust SDK don't help Go/Python/JS/Ruby/Java users. Epic 26 closes this gap.

4. **Surprise wins are real.** The 64KB zero-copy (+227%) and tail latency (-91%) improvements were the biggest actual wins — neither was the primary target. Performance optimization is empirical, not theoretical.

5. **Tradeoffs must be acknowledged.** +79% memory and +362% disk I/O are real costs. The RocksDB write buffer sizes may need revisiting if memory becomes a constraint.
