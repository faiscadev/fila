## Successes

1. **Sixteenth consecutive epic at 100% completion** — 2/2 stories delivered. Versioning policy formalized in `docs/compatibility.md`. All 6 SDKs now have TLS + API key auth support, closing the critical gap identified in Epic 15.

2. **Smart epic reshaping** — Original 3-story plan reduced to 2. Story 16.3 (stability release branches) dropped as premature — no users to serve backport workflows pre-1.0. Story 16.1 slimmed to docs-only (no GetServerInfo RPC, no --version flags). Scope discipline.

3. **Epic-review process caught critical security bugs** — When retroactive PRs were created for the 5 external SDKs, Cubic found 28 issues (5 P1s). Key P1: silent TLS downgrade when mTLS options provided without CA cert — present in all 5 SDKs. The epic-review workflow's compensating controls proved their value.

4. **System trust store support added as review-driven scope addition** — All 6 SDKs now support simple `tls: true` / `with_tls()` without needing to provide CA certificates. Better developer UX for publicly-signed servers.

5. **23 new tests** — Auth integration tests across SDKs, TLS activation checks, partial mTLS validation.

## Challenges

1. **External SDK changes pushed directly to main without PRs** — The biggest process gap of any epic. Security-sensitive auth/TLS code was pushed directly to main in all 5 external SDK repos — no pull requests, no Cubic review, no human review. The execute-epic workflow has no gate for external repo changes, so the dev agent took the fastest path. This is a structural gap, not an individual failure.

2. **No story spec file for Story 16.2** — Despite the CLAUDE.md rule about mid-epic story additions requiring spec files, no story spec was created for 16.2. The story existed in epics.md with full ACs but had no implementation-artifacts file, making AC verification harder during epic review.

3. **28 Cubic findings required retroactive cleanup** — Creating retroactive PRs (revert from main, create branch, open PR, run Cubic, fix findings, re-merge) is far more expensive than doing it right the first time. Prevention > cure.

## Key Insights

1. **Multi-repo workflow gaps are the #1 process risk** — The execute-epic workflow assumes all work happens in the main repo. External repo changes escape all quality gates (Cubic, PR review, epic-review verification). Now encoded in CLAUDE.md.

2. **Epic-review workflow saved this epic** — Without the retroactive PR process, 5 P1 security issues (silent TLS downgrade) would have shipped to all SDKs. The workflow created during Epic 5 retro proved its value in a high-stakes scenario.

3. **Silent TLS downgrade is a universal SDK bug pattern** — All 5 external SDKs had identical bug: providing mTLS cert/key options without a CA cert silently fell back to plaintext connection. Should be a standard test case template for any future SDK TLS work.
