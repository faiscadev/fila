## Successes

1. **Eleventh consecutive epic at 100% completion** — Full Fila roadmap delivered. 11 epics, every one at 100%.

2. **Dev-agent/human task split worked cleanly** — Story 11.2 explicitly labeled tasks as "dev-agent" (code fixes: duplicate YAML block, repo URL, crate metadata) vs "human (Lucas)" (registry accounts, secrets, GPG keys, workflow triggers). Zero ambiguity, no blocking between roles.

3. **Bleeding-edge pipeline worked first try** — All 5 jobs passed on first feature branch trigger. Epic 10's engineering was solid.

4. **Pre-existing bugs caught during verification** — Duplicate `publish-crates` block in release.yml would have silently published only 2 crates instead of the full chain. Workspace repository URL pointed to `faisca/fila` instead of `faiscadev/fila`. Both caught and fixed before publishing.

5. **All 5 SDK CIs running real integration tests** — Not silently skipping. The CLAUDE.md "Integration Tests Must Actually Run" rule (from Epic 9 retro) and "CI Workflow Verification" rule (from Epic 10 retro) both proved their value.

6. **6 package registries publishing successfully** — crates.io (fila-proto, fila-sdk, fila-cli), Go module tags, PyPI, npm, RubyGems, Maven Central.

7. **Previous retro action items: 5/5 completed (100%)** — Encode-or-drop validated for eleventh consecutive epic.

## Challenges

1. **Maven Central publishing significantly more complex** — Signing plugin, POM developers/scm sections, migration from dead OSSRH endpoint to Central Portal OSSRH Staging API, GPG key generation + keyserver upload, 4 separate secrets. Every other registry was straightforward by comparison.

2. **External dependency surprise** — fila-python grpcio 1.78.1 was yanked from PyPI. Required downgrading version check to 1.78.0. Unpredictable external issue.

3. **Story ACs drifted from reality** — AC7 said fila-server should be on crates.io, but Lucas decided to set it to `publish=false` (too many build dependencies for casual users). AC4 said `@fila/client` but actual npm package is `fila-client`. Code review caught both.

## Key Insights

1. **Explicit agent/human task ownership eliminates ambiguity** — When story definitions label each task with "dev-agent" vs "human (Lucas)", there's zero confusion about who owns what. This pattern should be used for any future epic with operational tasks.

2. **Operational verification epics are valuable** — Epic 10's untriggered pipelines became Epic 11's verified infrastructure. Building automation is not the same as verifying automation works.

3. **AC updates should happen at decision time** — When Lucas decided fila-server shouldn't be on crates.io, the AC should have been updated then. Code review caught it, so the safety net worked, but earlier is better.
