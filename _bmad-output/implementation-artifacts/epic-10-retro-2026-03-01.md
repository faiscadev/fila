## Challenges

1. **Pipelines built but never actually fired** — All CI workflows (bleeding-edge.yml, SDK publish workflows) were created but never triggered. Story 10.1 Task 4 ("verify end-to-end") was explicitly marked incomplete with the note "requires merge to main." Lucas flagged this: "is this really all published? or did we just build the pipelines that theoretically publish them." The answer was the latter.

2. **Post-merge operational work not explicitly tracked** — Secrets configuration, DNS setup, first registry publishes — these are all required for the pipelines to actually work but weren't part of any story's deliverables.

3. **Definition of "done" for CI workflows was too loose** — Building a workflow file is not the same as testing it. Lucas's position: "it should find a way to run it, even if that means changing the workflow so it triggers in the branch then change back."

## Key Insights

1. **CI workflows must be smoke-tested before marking done** — Temporarily broaden the trigger to fire on the feature branch, verify it works, then scope it back. This is the CI equivalent of running your unit tests. Now encoded in CLAUDE.md.

2. **Building automation is not the same as verifying automation works** — The pipelines are code. Untriggered pipelines are untested code.

3. **Operational work needs explicit tracking** — Secrets, DNS, registry config are not engineering tasks but they're prerequisites for the engineering to matter. Tracked in MEMORY.md as a checklist.

4. **Encode-or-drop continues to hold at 100%** — Ten epics, every encoded action item completed. The pattern is validated beyond doubt.
