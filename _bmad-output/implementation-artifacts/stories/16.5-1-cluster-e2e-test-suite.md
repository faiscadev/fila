# Story 16.5.1: Cluster E2E Test Suite

Status: in-progress

## Story

As an operator,
I want blackbox e2e tests proving cluster failover, leader routing, and replication work end-to-end,
so that I can trust multi-node deployments in production.

## Acceptance Criteria

1. **Given** the fila-e2e test suite and a 3-node cluster
   **When** a cross-node lifecycle test executes
   **Then** enqueue on node A, consume on node B, ack on node C completes successfully

2. **Given** a 3-node cluster with a queue leader
   **When** the leader node is killed
   **Then** a new leader is elected, the consumer reconnects, and zero messages are lost

3. **Given** a non-leader node receives a request
   **When** the request requires leader forwarding
   **Then** the client gets the correct response transparently

4. **Given** a node that crashed and restarted
   **When** the node rejoins the cluster
   **Then** it catches up from the Raft log and becomes eligible for leadership

5. **Given** a 3-node cluster with queued messages
   **When** `fila stats` is run on any node
   **Then** cluster-wide aggregated counts are returned

6. **Given** the CI pipeline
   **When** cluster e2e tests are included
   **Then** CI runs cluster e2e tests (extended `e2e.yml` or new workflow)
   **And** tests have reasonable timeouts accounting for Raft election (10s failover window per NFR23)

## Dev Notes

### Architecture Context

Clustering uses **Raft-per-queue** via `openraft`. Key components:
- `ClusterManager` (`crates/fila-core/src/cluster/mod.rs`) — starts meta Raft + cluster gRPC service
- `MultiRaftManager` (`crates/fila-core/src/cluster/multi_raft.rs`) — one Raft group per queue
- `ClusterHandle` — request forwarding, leader tracking, cached gRPC clients
- `ClusterConfig` (`crates/fila-core/src/broker/config.rs`) — `enabled`, `node_id`, `peers`, `bind_addr`, `election_timeout_ms`, `heartbeat_interval_ms`

**Port layout**: Client gRPC on `server.listen_addr` (default 5555), cluster gRPC on `cluster.bind_addr` (default 5556). Each test node needs both ports.

**Peer bootstrapping**: Node 1 uses `peers = []` (bootstraps single-node), nodes 2+ use `peers = ["127.0.0.1:<node1_cluster_port>"]` to join.

### Existing E2E Patterns to Follow

- `crates/fila-e2e/tests/helpers/mod.rs` — `TestServer` struct: spawns binary, polls TCP for readiness, drain stderr, RAII cleanup via `Drop`. `kill_and_take_data()` + `restart_on()` for crash/restart.
- `crates/fila-e2e/tests/lifecycle.rs` — SDK-based enqueue/consume/ack pattern.
- `crates/fila-e2e/tests/tls.rs` — custom config file generation + `TestServer::from_parts()`.

### TestCluster Design

Create a new file `crates/fila-e2e/tests/helpers/cluster.rs` and re-export from `helpers/mod.rs`. Pattern:

```rust
pub struct TestCluster {
    nodes: Vec<Option<TestServer>>,
    data_dirs: Vec<Option<tempfile::TempDir>>,
    client_ports: Vec<u16>,
    cluster_ports: Vec<u16>,
}
```

Each node gets its own `fila.toml` with cluster section. Node 0 bootstraps (`peers = []`), others join via node 0's cluster port. Use `free_port()` for both client and cluster ports.

`wait_for_leader` should poll via CLI `fila queue stats <queue>` or SDK, checking that a leader is reported. Timeout at 15s (generous for Raft election in test env).

### Critical Timing Considerations

- Raft election timeout is 1000ms by default. In tests, consider lowering `election_timeout_ms` and `heartbeat_interval_ms` for faster failover.
- After killing a leader, allow up to 10s (NFR23) for new leader election.
- After node restart, allow time for Raft log replay before asserting state.
- Use `tokio::time::sleep` or polling loops with timeout for all async waits.

### Existing Cluster Unit Tests

`crates/fila-core/src/cluster/tests.rs` has `TestNode` + `bootstrap_cluster()` + `wait_for_leader()`. These are unit tests using internal APIs. E2E tests must use only SDK + CLI (blackbox).

### CI Notes

Existing `e2e.yml` already builds workspace and runs `cargo nextest run -p fila-e2e`. Cluster tests will run as part of the same crate. No new binary dependencies needed — just longer timeouts. Consider adding `--test-threads=1` for cluster tests if port conflicts arise, or rely on `free_port()`.

If tests are slow (multiple Raft elections), consider a separate CI job or test filter. The existing CI job should work as-is since cluster tests are in the same crate.

### Testing Standards

- All tests async (`#[tokio::test]`)
- Use SDK client for data operations (enqueue/consume/ack)
- Use CLI for admin operations (queue create, stats)
- Blackbox only — no internal crate imports
- Each test self-contained: start cluster, run scenario, assertions, cleanup via Drop
