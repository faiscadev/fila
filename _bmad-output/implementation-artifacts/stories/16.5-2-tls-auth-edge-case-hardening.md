# Story 16.5.2: TLS & Auth Edge Case Hardening

Status: done

## Story

As a security-conscious operator,
I want comprehensive TLS and auth edge case tests,
so that security bypass patterns (like silent TLS downgrade) are caught automatically.

## Acceptance Criteria

1. **Given** the fila-e2e test suite
   **When** mTLS with a valid client certificate is tested
   **Then** the server validates the client cert and the connection succeeds

2. **Given** mTLS is required (ca_file set on server)
   **When** a client connects without a client certificate
   **Then** the server rejects the connection

3. **Given** TLS enabled on server
   **When** a plaintext client connects
   **Then** the connection is rejected (already tested — verify no regression)

4. **Given** an expired server certificate
   **When** a client connects
   **Then** the connection is rejected with a clear error

5. **Given** auth is enabled and a key is revoked
   **When** the revoked key is used on the next request
   **Then** the request is rejected immediately

6. **Given** auth is enabled and a permission is removed from a key
   **When** the key attempts the previously-authorized operation
   **Then** the operation is rejected immediately

7. **Given** auth is enabled with a bootstrap key
   **When** the bootstrap key attempts data operations (produce/consume)
   **Then** the operation fails — bootstrap key can only create keys (admin-only)

8. **Given** a superadmin key is revoked
   **When** the revoked superadmin attempts any operation
   **Then** all access is denied

9. **Given** these are universal patterns
   **When** test patterns are established
   **Then** a documented TLS test checklist exists (in test comments) listing mandatory scenarios

## Dev Notes

### Existing TLS Test Patterns

`crates/fila-e2e/tests/tls.rs` (161 lines) has 3 tests:
- `tls_valid_cert_connects_successfully` — self-signed cert via `rcgen`, server config with `[tls]` section
- `tls_plaintext_client_is_rejected` — plaintext HTTP/2 against TLS server
- `tls_wrong_ca_cert_is_rejected` — wrong CA cert fails verification

Pattern: uses `rcgen::CertificateParams` to generate certs, writes PEM files to temp dir, spawns server with config pointing to cert files, uses `ConnectOptions::new(&addr).with_tls_ca_cert(ca_pem)`.

### mTLS Configuration

Server config for mTLS requires `ca_file` in `[tls]` section:
```toml
[tls]
cert_file = "server.crt"
key_file = "server.key"
ca_file = "ca.crt"    # Presence enables client cert verification
```

Client connects with `ConnectOptions::new(&addr).with_tls_ca_cert(ca_pem).with_tls_client_cert(cert_pem, key_pem)`.

Config struct: `TlsParams { cert_file, key_file, ca_file: Option<String> }` in `crates/fila-core/src/broker/config.rs`.

### Generating mTLS Cert Chain with rcgen

```rust
// 1. Create CA
let ca_params = CertificateParams::new(vec!["Test CA".to_string()]);
let ca_cert = Certificate::from_params(ca_params).unwrap();

// 2. Create server cert signed by CA
let mut server_params = CertificateParams::new(vec!["127.0.0.1".to_string()]);
server_params.is_ca = IsCa::NoCa;
let server_cert = Certificate::from_params(server_params).unwrap();
let server_cert_pem = server_cert.serialize_pem_with_signer(&ca_cert).unwrap();

// 3. Create client cert signed by same CA
let mut client_params = CertificateParams::new(vec!["test-client".to_string()]);
client_params.is_ca = IsCa::NoCa;
let client_cert = Certificate::from_params(client_params).unwrap();
let client_cert_pem = client_cert.serialize_pem_with_signer(&ca_cert).unwrap();
```

### Bootstrap Key Behavior

The bootstrap key (`CallerKey::Bootstrap`) is hardcoded to bypass storage lookup. It has superadmin privileges — which means it CAN do data operations unless explicitly restricted. Need to verify actual behavior:
- In `crates/fila-server/src/auth.rs`, the middleware returns `CallerKey::Bootstrap` for bootstrap key
- Hot path service checks `ValidatedKeyId` for produce/consume permissions
- Bootstrap key maps to `CallerKey::Bootstrap`, and `is_superadmin` → `true` for `Bootstrap` variant
- So bootstrap key actually HAS produce/consume permissions (superadmin bypass)

**Important:** If bootstrap key is effectively superadmin, AC 7 (bootstrap can't do data ops) may not match current behavior. Need to verify and adjust test expectations accordingly. The AC may need to test that bootstrap key CAN do all ops (since it's superadmin).

### Auth Module Internals

- `crates/fila-core/src/broker/auth.rs` — `CallerKey`, `Permission`, `ApiKeyEntry`, `has_permission()`
- `crates/fila-core/src/broker/mod.rs` — `validate_api_key()`, `revoke_api_key()`, `check_permission()`
- `crates/fila-server/src/auth.rs` — Tower middleware, `ValidatedKeyId` injection
- Key revocation: `revoke_api_key(key_id)` deletes from RocksDB. `validate_api_key()` does O(n) scan, so revoked key is immediately not found.
- Permission update: `set_acl(key_id, permissions)` replaces all permissions. Next `check_permission()` call reads fresh from storage.
