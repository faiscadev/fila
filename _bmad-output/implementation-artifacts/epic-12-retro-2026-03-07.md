All would have caused failures if merged. Code review continues to be a critical safety net.

## Challenges

1. **Story 12.3 attracted significant Cubic findings** — 9+1 issues across 2 review cycles. Python code seems to require more review attention than Rust (no compiler safety net). All findings were addressed, but the iteration cost was notable.

2. **Story 12.1 code review bugs** — 4 bugs that would have been runtime failures. The code was syntactically correct but semantically wrong (wrong address format, wrong Lua wrapper pattern). These are the hardest bugs to catch — they compile fine but fail at runtime.

3. **Story 12.1 status tracking mismatch** — Story file shows "review" status but sprint-status.yaml correctly shows "done". Minor tracking inconsistency in the story file itself.

## Key Insights

1. **Blackbox system benchmarks need different tooling than micro-benchmarks** — criterion is wrong for system-level benchmarks. Custom harness with server lifecycle management (spawn, warmup, measure, teardown) is the right approach. The `harness = false` pattern worked well.

2. **Rust binaries > shell scripts for CI tooling** — Compare and aggregate tools as Rust binaries in fila-bench are testable (13 unit tests), type-safe, and avoid bash parsing fragility. This pattern should be replicated for future CI tooling needs.

3. **Python is the lingua franca for cross-broker benchmarking** — Every broker has a well-maintained Python client. Writing Rust clients for Kafka, RabbitMQ, and NATS would have been massive scope for minimal benefit.

4. **Epic 12 benchmark suite enables Epic 13 validation** — Story 13.5 ACs explicitly reference "benchmark comparison using the suite from Epic 12" for NFR validation. The benchmark infrastructure is a prerequisite, not just a nice-to-have.
