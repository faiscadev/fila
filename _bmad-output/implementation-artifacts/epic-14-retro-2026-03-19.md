## Successes

1. **Fourteenth consecutive epic at 100% completion** — 6/6 stories delivered. 14 epics, every one at 100%.

2. **Zero debug struggles across 6 stories of distributed systems work** — Stories 14.2 and 14.4 explicitly note "None" in debug log references. Shipping Raft consensus, multi-Raft management, leader failover, and node recovery on first pass without debugging marathons is exceptional.

3. **Epic 13 seams worked exactly as designed** — `QueueRouter` (13.2) served directly as the placement table in 14.2. `apply_mutations()` (13.1) was the state machine application point in 14.1. `DRR key-set scoping` (13.2) required zero code changes when per-group scheduling arrived. The thin seams guided exactly where to cut.

4. **Deferred task handoff was clean** — Story 14.2 explicitly flagged Tasks 5 & 6 as deferred to Story 14.3. Story 14.3 picked them up as Tasks 1 & 2 with precise references. No regression, no confusion, no re-implementation.

5. **Code review caught real bugs in every story** — 14.1: panic on config validation, missing TOML test. 14.2: stale RocksDB data on queue re-creation (correctness bug), proto field number collision. 14.3: tempdir leak in test harness, Box required for large enum variant. None were style nits — all were correctness, safety, or resource issues.

6. **Single-node mode discipline** — Every story had an explicit "verify single-node mode unchanged" task with a test count checkpoint. This is why zero regressions exist across 6 stories of Raft infrastructure addition.

7. **openraft integration decisions were sound** — `OnceLock<Broker>` for chicken-and-egg startup, `MetaStoreEvent` channel decoupling Raft state machine from queue lifecycle, key-prefix isolation instead of per-queue column families. These are the kind of decisions that prevent future architectural regret.

## Challenges

1. **Story 14.6 had no story spec file** — The protobuf serialization work (PR #62) was added as a 6th story mid-epic. It was tracked in sprint-status.yaml and executed via proper branch/PR, but no story spec file exists in `_bmad-output/implementation-artifacts/stories/`. Anyone reading story files gets an incomplete picture of Epic 14. This is now encoded in CLAUDE.md.

2. **openraft v1 adaptor pattern is tech debt** — The v2 `RaftStorage` traits are behind an unstable feature gate. Story 14.1 completion notes flag this explicitly: `RaftStorage + Adaptor` used because v2 is not stable. The adaptor adds indirection. When openraft v2 stabilizes, a migration will be needed.

3. **`find_last_log_id` scans all entries** — Story 14.1 notes this uses `usize::MAX` scan. Acceptable for the meta Raft group (small log), but should be replaced with a reverse iterator before log sizes grow. Not a blocker — a known limitation.

## Key Insights

1. **Thin seams pay off when the dependent epic arrives** — Epic 13's 2 prep stories saved weeks of rework in Epic 14. `QueueRouter`, storage trait, DRR key-set scoping — all integrated directly without code changes. Research-driven epic reshaping and deliberate thin seams are the right strategy.

2. **Code review is the primary safety gate, not tests alone** — 6 real bugs caught across 3 stories that passed all tests. The Rust compiler and test suite couldn't catch: a panic on config validation, a RocksDB stale data correctness bug, a test harness resource leak, a proto field numbering collision. Human (or agent) code review is irreplaceable.

3. **Deferred tasks need explicit story-file references** — The 14.2→14.3 deferred task handoff worked because the story files were clear ("deferred to Story 14.3" with explicit task references). This is the correct pattern for scope management across a story chain.

4. **Mid-epic scope additions need story spec files** — Story 14.6 was real work that shipped, but the only records are git history and sprint-status. Future developers and agents relying on story files get an incomplete picture. Encoded in CLAUDE.md.
