## Epic Summary

Epic 21 replaced the naive benchmarking infrastructure with statistically rigorous measurement methodology. Key deliverables:

1. **HdrHistogram foundation** (21.1) — replaced Vec<Duration> sort-based percentiles with hdrhistogram crate, duration-based collection (30s), extended percentile reporting (p50/p95/p99/p99.9/p99.99/max), histogram serialization for cross-run aggregation
2. **Competitive benchmark overhaul** (21.2) — concurrent produce/consume pattern, timestamp-in-payload for true e2e latency, 30s measurement duration, disk I/O measurement, 3-run aggregation
3. **Open-loop load generation** (21.3) — latency-under-load at 50%/80%/100% capacity, backpressure ramp, queue-depth impact, coordinated omission correction
4. **Failure-path benchmarks** (21.4) — nack storm, DLQ routing overhead, poison pill isolation, Jain's Fairness Index
5. **CI visualization** (21.5) — github-action-benchmark integration, polarity-aware metric splitting, GAB JSON format

## What Went Well

### Research-Driven Spec Eliminated Scope Surprises
The benchmarking methodology research pre-identified all 12 measurement gaps with priority. Every story mapped directly to research findings. Zero mid-epic scope changes, zero discovery during implementation.

### Parallel Story Structure
Stories 21.2–21.5 all depend only on 21.1. The independence allowed parallel branch development with minimal merge conflicts (only system.rs registration order). This is the ideal epic shape for throughput.

### HdrHistogram Pattern Reused Cleanly
The histogram serialization → base64 → JSON → deserialize → merge pattern from 21.1 was reused in 21.2 (competitive benchmark aggregation) and 21.5 (GAB format generation) without modification. Good abstraction boundary.

### Cubic Caught Real Bugs (Story 21.1)
All 4 Cubic findings in 21.1 were genuine logic bugs:
- Histogram dedup by value would silently drop identical runs
- Sub-microsecond samples not clamped to minimum
- Unbounded collection loop if samples never reached threshold
- Stale loop condition documentation

None were style nits. Code review remains the primary quality gate.

### Zero Debug Struggles
All 5 story spec files report "Debug Log References: None." The benchmarking domain is well-understood and the implementation followed the research spec closely.

## What Didn't Go Well

### Competitive Benchmark Untestable in CI
Story 21.2 requires Docker containers for Kafka, RabbitMQ, and NATS. The concurrent latency pattern and 3-run aggregation are verified manually only. No automated regression detection for competitive benchmarks.

### Long-Running Benchmarks Gated Behind Env Vars
Stories 21.3 (open-loop) and 21.4 (failure-path) take minutes to run and are gated behind `FILA_BENCH_OPENLOOP=1` and `FILA_BENCH_FAILURE_PATHS=1`. They don't run in CI, so correctness regressions would be caught only during manual benchmark runs.

### Post-Merge Bug in 21.5
Queue creation in competitive benchmarks failed on 2nd+ runs because the Fila server persists between 3-run aggregation cycles. Fixed in commit d2cb526 by tolerating "already exists" errors. This should have been caught before merge — the 3-run pattern was known from 21.2.

### No Tests for 3 of 5 Stories
Stories 21.2, 21.3, and 21.4 have zero automated tests. The justification (Docker dependency, long runtime) is valid but leaves 60% of the epic's code without regression protection. Only 21.1 and 21.5 have meaningful test coverage.
