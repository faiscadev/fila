## Successes

2. **Cubic caught real backward compatibility issue** ‚Äî PR #84 added `preferred_leader` field to `CreateQueueGroup`. Fixed with a fallback to smallest-member-ID behavior when `preferred_leader` is 0 or invalid.

3. **Clean separation of concerns across stories** ‚Äî 17.1 (error clarity + ack/nack performance), 17.2 (consume leader hint + SDK reconnect), 17.3 (queue-to-node assignment) had no overlapping code changes. Each story was self-contained.

## Key Insights

1. **Backward compatibility for Raft log entries is critical** ‚Äî Any field added to `ClusterRequest` variants that participate in the Raft log (like `CreateQueueGroup`) must handle the absence of that field gracefully. Serde `#[serde(default)]` and proto3 defaults-to-0 both need to be accounted for. Cubic caught this ‚Äî it would have been a production bug.

2. **Queue-to-node assignment is simple but effective** ‚Äî The load-aware strategy (count current leaderships, pick least-loaded) is straightforward but covers the common case well. Complex rebalancing (moving leadership of existing queues) was correctly deferred ‚Äî it's unnecessary until operators report uneven load after node failures.

3. **Task 3 (expose mapping) was already done** ‚Äî `leader_node_id` was already in both `QueueInfo` and `GetStatsResponse` from Epic 14 work. Story ACs should be verified against current code before implementation starts to avoid redundant work.
